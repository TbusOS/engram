## 0. 一句话目标

让 engram 成为业界第一个**同时**满足以下两条的开源记忆系统:

1. **越用 → 越准**:检索命中率随使用次数单调上升,可在公开 benchmark 上量化
2. **越用 → 越省**:用户人均录入摩擦 / 维护成本随使用次数单调下降,可在 wisdom report 里画出曲线

业界 mem0 / Letta / Zep 偏 1,Obsidian / basic-memory 偏 2,无一同时做到。这是 engram 唯一值得做的卖点,也是判断每个新功能 "做不做" 的唯一标尺:**这条新功能能让 6 条 wisdom 曲线之一往好的方向走吗?不能就不做**。

## 1. 6 条 wisdom 曲线 — 不是感觉,是数字

每条曲线在 `engram wisdom report` 里可绘制,曲线方向不对就是设计错。

| ID | 曲线 | 含义 | 数据来源 | 当前 |
|---|---|---|---|---|
| **C1** | 检索命中率 | 第 N 次类似任务 context pack 命中正确 asset 的比例上升 | `usage.jsonl` event_type=loaded + LLM 回写命中标注 + LOCOMO/LongMemEval 跑分 | 缺 |
| **C2** | 任务复发耗时 | 第 2 次做相似任务比第 1 次快 X%(AgentFactory Batch 1 vs 2 思路) | task_hash + workflow run journal 时长 | 缺 |
| **C3** | 用户写入摩擦 | 单条 memory 平均录入耗时 / 字符数 / 命令次数下降 | CLI 操作埋点 + `engram memory quick` 使用率 | 缺 |
| **C4** | mandatory 误伤率 | 用户主动 dismiss / override mandatory 的次数下降 | `journal/consistency.jsonl` + ambiguous_conflict 记录 | 缺(依赖 issue #6 落地) |
| **C5** | 冗余率 | 同主题重复 memory 的比例下降(Consistency 自动合并) | Phase 2 body-hash 检测计数 + dedup 后 asset 总数 | 部分(Phase 2 done,evaluator 未自动建议合并) |
| **C6** | confidence 校准 | high-confidence asset 的实际命中率 ≈ confidence 数值 | usage.jsonl 事件证据(issue #9 模型) | 缺(必须先 issue #9 落地) |

**根本机制**:`~/.engram/journal/usage.jsonl` 不是 confidence 一项的事件源,**是 6 条曲线的统一数据脊柱**。issue #9 提到的事件证据模型必须升级为 engram 整个 "学习神经系统" 的中枢。

## 2. 三引擎分工 — 缺一个就没闭环

| 引擎 | 职责 | 学习信号 | 输出 | 当前 |
|---|---|---|---|---|
| **Relevance Gate**(T-40 done) | 选当前任务相关 asset | usage.jsonl 命中反馈 | packed context | 6 stage 跑通,Stage 3 hack(待 RRF) |
| **Consistency Engine**(T-46 部分 done) | **不退化**:发现冲突 / 冗余 / 过期 | 全 store 扫描 | archive / dismiss / merge 提案 | Phase 1/2 done,Phase 3/4 stub |
| **Evolve Engine**(**未建**) | **主动变好**:从 archive + journal 学规律,生成 asset 改进提案 | usage.jsonl + Consistency 历史 + Wisdom Metrics | split / promote / demote / fork 提案 | 0% |

Evolve Engine 不存在 = "越用越好用" 永远停在 "不退化" 而不是 "主动变好"。这是 Darwin Gödel Machine 思想没落地的代价(详见 `reference_competitor_architecture_notes.md`)。

## 4. 关键设计原则

1. **永远不要 "为了好用牺牲学习信号"**:`memory quick` 让录入更省事,但事件证据照样写。降摩擦的方向是**自动派生**(从 git 推 task_hash),不是**不记**。
2. **三引擎分工要硬切**:Relevance 不写 store,Consistency 不主动改 asset,Evolve 只生提案不自动 mutate。SPEC §1.2 原则 4 "engram 永不擅自改写人类内容" 是地基,Evolve Engine 也要遵守。
3. **可量化的进步 > 任何感性宣传**:GitHub README、官网首页、对外文章一律以 6 条曲线 + LOCOMO 数字为论据,删除任何 "据说更智能" / "感觉更聪明" 的措辞。
4. **核心使命对齐**(每个 P0 都过这个筛):
   - 让 LLM 切换平台行为不变(issue #6 ambiguous + issue #9 confidence 派生 cache)
   - 让本地数据可移植到任何机器(issue #1 init adopt + issue #4 graph.db canonical URI)
   - 让 store 越用越聪明(usage bus + 6 条曲线 + Evolve Engine)

## 5. 不做的事

- **不引入新存储引擎**:graph.db 仍是 SQLite,markdown 仍是地基,不引入 Postgres / Neo4j / 向量数据库专用产品。
- **不做云托管**:engram 是本地优先 + 可移植,任何 "engram cloud" 想法属于另一个项目。
- **不做 LLM-side 实时拦截**:Consistency Engine 仍是 post-hoc(扫 → 提案 → 人决定),不接管 LLM 输出流。
- **不为兼容性留 5 个月以上的历史 shim**:v0.2.1 SPEC-AMEND 一次性迁移,migrate 命令把老 store 直接转成新 schema,不做长期双轨。
