# adapters/ — thin prompt templates for each LLM tool

**Status**: skeleton only (M4 not started)

Adapters tell a specific LLM tool where to find `.memory/` and how to use it. Each adapter is a tiny template — typically one markdown file — that gets rendered into the user's project by `engram adapter <tool>`.

## Design invariants

1. **Adapters are thin.** An adapter MUST NOT reimplement CLI logic. If an adapter grows past ~100 lines of template, something is wrong.
2. **Adapters are one-way.** `engram adapter X` generates files. Users can edit the generated files (marker-wrapped regions preserve user changes).
3. **Adapters share one `.memory/`.** Enabling multiple adapters in the same project is supported. The underlying store is untouched.
4. **Adapters are optional.** A user who doesn't want any adapter can still paste `.memory/MEMORY.md` into any LLM manually.

## Template structure

Each adapter directory contains:

```
adapters/<tool-name>/
├── README.md                  # what this adapter does, how to use it
├── <output-file>.tmpl         # the main template
└── <optional extras>          # e.g. skill files, hook snippets for tools that support them
```

Templates use simple `{{variable}}` placeholders. Variables available at render time:

| Variable | Meaning |
|----------|---------|
| `{{project_name}}` | basename of the project root |
| `{{memory_dir}}` | path to `.memory/` relative to project root |
| `{{date}}` | ISO date at render time |
| `{{engram_version}}` | spec / CLI version |
| `{{methodology_url}}` | canonical URL for `METHODOLOGY.md` (fetched so LLM can read it) |

## Marker convention

Generated content is wrapped in markers so users can hand-edit without losing regenerate-safety:

```markdown
<!-- BEGIN engram -->
(engram-managed content, regenerated by `engram adapter <tool>`)
<!-- END engram -->

(user-authored content below the marker — engram never touches this)
```

`engram adapter <tool> --regenerate` replaces only the content between `BEGIN engram` and `END engram`.
