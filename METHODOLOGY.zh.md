# engram 记忆写作方法论 (v0.2)

**版本**: 0.2 — 配套 SPEC v0.2 + DESIGN v0.2
**读者**: 正在写记忆的 LLM；其次是希望理解 LLM 记忆纪律的人
**最后更新**: 2026-04-19

---

## §1. 核心原则

**记忆是长期资产，不是便签本。**

一条好的记忆在六个月后仍然有用——可能被另一个 LLM 读，可能被从未见过当时那次对话的新成员阅读。写的时候心里放着这样的读者。

三条推论由此而来：

**1. 写一次，写对。** 或者先写成草稿、用 supersede 链逐步演化——而不是"以后再改"。`draft` 生命周期状态正是为此而设：如果不确定，先标记为草稿，但要承诺最终把它解决，而不是让 store 长期处于悬而未决的状态。

**2. 足够通用，能够适用。** 如果一条记忆读起来像"在那次对话里我们商量了 X"，它可能太窄了。测试标准：这条记忆能在完全不知道原始对话的新会话中有意义地改变行为吗？如果不能，它不属于 store。

**3. 足够具体，可以执行。** "小心文件操作"不是规则。"`rm -rf` 或 `git push --force` 之前先询问用户"才是。模糊的记忆通过了写入时的检查，却在加载时失败——相关性闸门对它们评分很低，而确实加载了它们的 LLM 也找不到任何可执行的内容。

这三条推论相互叠加。过于狭窄的记忆六个月后毫无用处。过于模糊的记忆浪费上下文预算。认真写的记忆会惠及每一个未来的会话、每一个未来的 LLM、每一个继承这个 store 的未来成员。

## §2. 什么时候写记忆

当你学到的东西同时满足以下三个条件时，才写记忆：

1. **对未来的对话有价值** — 不只是这一次。问自己："如果下个会话从零开始，知道这件事会改变什么吗？"
2. **无法从代码、git 记录或公开文档中恢复** — 如果事实本身就在 commit message 或 README 里，一条指向那里的 `reference` 记忆就够了，不要重复内容。
3. **忘掉会导致重复犯错** — "意外"信号。任何让你感到意外的事情都是候选。任何引发纠正的事情都是强候选。

### 立刻写，不要攒

用户纠正你、确认某个非直观选择是对的、或表达了长期偏好时，就在那一轮写记忆。攒一批再写看起来高效，但会丢失"为什么"——到一次长对话结束时，使那个偏好有意义的上下文已经压缩或丢失了。

写一条多余记忆的代价很低（一次 review 就能清理掉）。丢失一个信号的代价很高（行为回退、用户再次纠正、对话重演）。

### 纠正声音大，确认声音小。两种都值得写。

- 纠正："别那么干"——信号明显，容易抓到。
- 确认："对，就是这样"/"那次决定是对的"——信号安静，容易错过。
- 模式信号：如果两周后你有可能再次提出那个被拒绝的选项，就把拒绝和原因都存下来。

只存纠正会让 store 能避免过去的错误，但会慢慢偏离用户已经认可的做法。确认也要存。

### 用户明确要求时立即存

用户说"记住这个"或"存一下"时，当轮就写。除非范围真的不清楚（比如"这是项目级规则还是个人规则？"），否则不要反问。用户的明确指令已经足够授权。

## §3. 选对子类型

共有六种子类型。子类型表达的是**认识论状态** — 谁写的，怎么知道它为真。这和 scope 正交：同样一句话，人说出来是 `feedback`；LLM 从观察中推断出来，就是 `agent`。

### `user` — 关于人的事实

**何时写:** 你学到了关于这个人的信息 — 角色、技能、经验水平、工作背景、沟通偏好 — 会改变你与他交流的方式。

**必填 frontmatter:** `name`、`description`、`type: user`、`scope`

**Body:** 从 LLM 视角写第三人称自由散文（"该用户负责……""他们偏好……"）。2–8 段。无强制子节。`enforcement` 默认为 `hint`——用户信息是建议性的，不是强制规则。

**示例触发:** "我写 Go 已经十年了，刚开始学 Rust。" → `user` 记忆："用户有 10 年以上 Go 经验；Rust 对他们来说是新的——相应校准解释深度。"

---

### `feedback` — LLM 必须遵守的规则

**何时写:** 用户给出了直接规则，纠正了你的行为，或确认某个非直观做法是对的。

**必填 frontmatter:** `name`、`description`、`type: feedback`、`scope`、`enforcement`（此子类型必须显式声明，没有默认值）

**Body 结构:**

```
<一行规则陈述>

**Why:** <原因 — 通常是某次事故或明确偏好>

**How to apply:** <这条规则何时、何地生效；包含边界情况>
```

没有"Why"的规则会变成迷信。没有原因，未来的 LLM 无法对边界情况进行推理。

**Enforcement 级别:**
- `mandatory`（强制级）— 不可被下级 scope 覆盖；违反时 `engram validate` 报错。用于公司合规要求、安全策略、团队硬约束。
- `default`（默认级）— 可覆盖，但覆盖方必须声明 `overrides: <id>`。用于团队惯例、推荐做法。
- `hint`（建议级）— 可自由覆盖，无需说明。用于个人风格偏好。

**示例触发:** "不经过我确认不许 force push。" → `feedback` 记忆，`scope: user`，`enforcement: mandatory`。

---

### `project` — 进行中的工作事实

**何时写:** 做出了决策，设定了截止日期，从事故中总结了教训，浮现了干系人约束 — 任何关于进行中工作、无法仅靠代码或 git 历史恢复的事实。

**必填 frontmatter:** 同 `feedback`，去掉 `enforcement`（默认为 `hint`）

**Body 结构:**

```
<一行事实或决策陈述>

**Why:** <动机 — 约束、截止日期、干系人要求>

**How to apply:** <这对未来建议或计划有何影响>
```

**只用绝对日期。** 写入时把"下周四"转成 `2026-04-23`。相对日期在记忆被一个月后读到时毫无意义。事实有时间边界时加上 `expires:` — 一致性引擎会在日期过后把它浮出来 review。

**示例触发:** "2026-04-20 之后要冻结合入，为发布让路。" → `project` 记忆，加 `expires: 2026-04-21`。

---

### `reference` — 指向外部资源的指针

**何时写:** 用户指给你一个系统、面板、工单跟踪系统、内部 wiki 或权威外部代码库，你以后还会用到。

**Body:** 自由散文。涵盖三件事：如何定位资源、为什么它重要、什么时候去查它。遗漏"什么时候去查"的 `reference` 在相关性闸门里得分很低，该加载时往往不会加载。

**示例触发:** "在 Linear 项目 INGEST 里找 pipeline 的 bug。" → `reference` 记忆：系统名称、它跟踪什么、什么时候去看。

---

### `workflow_ptr` — 工作流指针

**何时写:** 一个常用的过程模式作为工作流资产存在于 `workflows/<name>/`，需要在 `MEMORY.md` 中浮现，让 LLM 能发现它，而不必把完整的 `workflow.md` 提前加载进上下文预算。

**必填 frontmatter:** `workflow_ref`（指向 `workflows/<name>/` 的路径）

**Body:** 1–3 段，涵盖：工作流做什么、什么时候用、预期结果是什么。不要复制步骤 — 那些在 `workflow.md` 里。Body 回答的是"我该加载这个工作流吗？"而不是"工作流全文是什么？"

**示例触发:** 团队有一个 `workflows/release-checklist/`，涵盖从分支冻结到 changelog 发布的每个步骤。添加一条 `workflow_ptr` 记忆，让 LLM 在会话开始、遇到发布相关任务时能发现它。

---

### `agent` — LLM 学到的启发式规则

**何时写:** 你在多个会话中注意到了一个有效的模式，但没有人明确说过它。来源是你自己的观察，不是用户指令。

**必填 frontmatter:** 同 `feedback`，加上 `source: agent-learned`（或具体的工作流版本引用），加上从 `{validated_count: 0, contradicted_count: 0}` 开始的 `confidence` 块。

**Body 结构:** 同 `feedback` — 一行启发式规则 + **Why:** + **How to apply:** — 但"Why"必须引用具体的观察结果。"在 5 次合并中用这个方法观察到成功"是合格的 Why。"这样感觉更整洁"不是。

**默认置信度低于 `feedback`。** 一致性引擎对 `agent` 记忆的审查比 `feedback` 更频繁。`agent` 记忆是假设；`feedback` 记忆背后有人的权威。

**示例触发:** 在五次部署中，提前 squash commit 消除了 CI 不稳定性之后，创建一条 `agent` 记忆，记录这个模式、其证据基础和已知的适用限制。

---

### 决策树：选哪种子类型？

按顺序执行：

1. 这是关于这个人的事实？ → `user`
2. 这是人说的或确认的规则？ → `feedback`（显式设置 `enforcement`）
3. 这是 LLM 从结果中推断的启发式，没有人明确说过？ → `agent`（加 `confidence` 块）
4. 这是关于进行中的工作 — 截止日期、决策、事故？ → `project`（用绝对日期）
5. 这是指向外部资源的指针？ → `reference`
6. 这件事有完整的、可执行的过程存在？ → 不是记忆 — 是工作流；用 `workflow_ptr` 来暴露它
7. 这是需要专门阅读的大段领域材料 — 多节、供参考？ → 不是记忆 — 是知识库文章；如有需要用 `reference` 记忆指向它

## §4. Scope 与 Enforcement 纪律

每条记忆都需要 `scope:`。写之前先想清楚。

### Scope 决策树

1. 这适用于你个人参与的每个项目？ → `user` scope
2. 这只适用于某一个团队或部门？ → `team` scope
3. 这适用于组织内的所有团队？ → `org` scope
4. 这只适用于当前这个项目？ → `project` scope
5. 这对你的归属层级之外的人也有用？ → 考虑创建或订阅一个 `pool`

**默认：`project` scope。** 从最本地开始，有需要再提升。只有当规则明显更宽泛时，才提升到 `user`、`team` 或 `org` — 当它能帮助多个项目、多个人或整个组织时。过早提升 scope 会给更高层级的上下文增加噪声。

**过度扩大 scope 的代价。** `user` scope 记忆会在该用户的每个项目会话中加载。`org` scope 记忆会在组织内每个成员的每个项目中加载。scope 越广，记忆就需要写得越细心，`limitations:` 就越重要，`**Why:**` 就越需要详细。

### Enforcement 级别（`feedback` 必填，其他可选）

**`mandatory`（强制级）** — "必须遵守；覆盖是校验错误。"

在以下情况写 `mandatory`：
- 存在合规或法律要求
- 安全策略在执行
- 团队没有任何合理例外的硬约束

不要把 `mandatory` 作为默认值。门槛是"正常工作中没有任何合理情形下应该绕过这条规则"。如果存在合理的边界情况，用 `default`。

**`default`（默认级）** — "应该遵守；有显式声明时可覆盖。"

在以下情况写 `default`：
- 这是团队的标准做法
- 偏离是合理的，但应该被记录在案
- 是强烈建议，不是硬约束

要覆盖 `default` 记忆，覆盖方的资产必须包含 `overrides: <id>`。

**`hint`（建议级）** — "这是偏好；自由覆盖。"

在以下情况写 `hint`：
- 这是个人风格偏好
- 这是一个弱建议
- 你想把规则留存在案，但对别人忽略它没有意见

### 冲突解决

当加载的上下文中两条记忆相互矛盾时：

1. `mandatory` 胜过 `default` 胜过 `hint` — 与 scope 无关
2. 在相同 enforcement 级别内，**越具体越赢**：`project > user > team > org`
3. pool 内容按其 `subscribed_at` 层级参与这一解析顺序
4. 仍然僵持时，LLM 在两条记忆都存在的情况下进行裁决；`engram consistency scan` 将情况标记出来供人工 review

**在没有明确操作人批准的情况下，永远不要覆盖 `mandatory` 记忆。** 如果你认为某条强制规则有问题，给用户写说明——不要悄悄绕过它。

## §5. 置信度维护

每条记忆跟踪四个证据字段：`validated_count`（验证次数）、`contradicted_count`（证伪次数）、`last_validated`（最近验证）、`usage_count`（调用次数）。这些构成 frontmatter 中的 `confidence` 块。

### 作为 LLM 的责任

**当你据某条记忆行动，现实证明它是对的：**

```bash
engram memory validate-use <id> --outcome=success
```

**当现实与它矛盾** — 用户纠正你，测试失败，实际行为与记忆预测的不同：

```bash
engram memory validate-use <id> --outcome=failure
```

**被动使用**（记忆加载进了上下文，但你没有特别据此行动）→ 不需要调用。

### 为什么重要

一致性引擎使用置信分（`confidence_score`）把过时和被证伪的记忆浮出来供人工 review。公式：

```
score = (validated_count - 2 × contradicted_count - staleness_penalty) / max(1, total_events)
```

没有结果信号，引擎无法区分好记忆和坏记忆。一条 `validated_count: 0`、`contradicted_count: 0`、`usage_count: 40` 的记忆，对引擎来说什么都说明不了。一条 `validated_count: 10`、`contradicted_count: 0` 的记忆，表明规则得到了充分支持。在结果发生时发出信号。

### 初始化新记忆

新创建的记忆 — 尤其是 `agent` 类型 — 从 `{validated_count: 0, contradicted_count: 0}` 开始。这是设计如此：它们作为假设起步，通过证据毕业。在大约 N=3 次成功使用而无证伪后，记忆进入 `stable`（稳定）生命周期状态。具体的 N 可配置；原则是稳定需要证据。

### 陈旧惩罚

公式根据 `last_validated` 距今多久施加陈旧惩罚：

- 90 天以内：无惩罚
- 90–365 天：0.3 惩罚
- 超过 365 天：0.7 惩罚

你不需要自己计算。结果发生时调用 `validate-use`，引擎负责评分。实际含义：一条两年前正确、但从未再验证的记忆会得分很低并浮出 review — 这是预期行为。

## §6. `limitations:` 字段

可选的 `limitations:`（限制）frontmatter 字段（字符串列表）是一个**诚实的边界声明** — 它告诉未来的读者这条记忆何时不适用。

有真实边界情况时使用它。不是作为对冲（"可能并不总是适用"），而是作为具体的范围说明：

```yaml
limitations:
  - 基准测试仅在 M1 Pro 上测量；x86_64 结果可能不同
  - 基于 Tokio v0.4.x 编写；v1.0 之后的 async 行为可能变化
  - 不适用于 hotfix 分支——那些有自己的流程
```

没有 `limitations:`，读者会过度使用规则。有了它，他们就知道什么时候该跳出去，而不需要从头推导例外情况。

**`limitations:` 尤其重要的场景：**

- `feedback` 记忆：有真实例外情况，你已经想清楚的规则
- `project` 记忆：在特定约束下做出的决策，这些约束不会永远成立
- `agent` 记忆：在有限上下文中观察到的模式——明确写出那个上下文

**一致性引擎的交互。** 如果一条记忆有很高的 `usage_count`、任何大于零的 `contradicted_count`、且没有声明 `limitations:`，一致性引擎会提议添加限制。它是在暗示：记忆的边界情况已经被观察到但没有被记录。接受或完善这个提议 — 不要在没有检查那些证伪是否揭示了真实的范围限制的情况下忽略它。

## §7. 跨仓收件箱纪律

在为某个仓库工作时，你可能发现另一个仓库的问题。跨仓传信器（`engram inbox send`）允许你向另一个仓库的收件箱发送结构化消息。

### 何时发送

- `bug-report`：B 仓库的 API 或代码存在影响你在 A 仓库工作的具体 bug
- `api-change`：你有改进 B 仓库 API 的具体提案
- `question`：你需要只有 B 仓库维护者才能提供的信息
- `update-notify`：你在 A 仓库做了 B 仓库的 LLM 需要知道的变更
- `task`：你在 B 仓库请求具体的工作

### 何时不发送

- 没有具体提案的模糊反应（"这个设计感觉不对"）
- 没有对正确性产生直接影响的风格偏好
- 可以从 B 仓库现有文档中得到答案的问题
- 已有一条相同的待处理消息的重复（发送前先检查）
- 属于 GitHub Discussions 的开放性设计讨论

### 消息纪律

包含 `related_code_refs` — 使用 `path/file.py:L42@sha` 格式，让接收方能直接导航到问题，而不需要猜测位置。在 `**How to resolve:**` 部分包含具体的解决路径："在第 42 行添加检查"是可执行的；"这里应该更好"不是。对大多数问题使用 `severity: warning`，只有 bug 导致数据丢失、安全风险或硬性失败时才用 `severity: critical`。

```bash
engram inbox send \
  --to=repo-b \
  --intent=bug-report \
  --severity=warning \
  --message="read_config() 在文件缺失时静默返回 {} — 应当抛出 FileNotFoundError。" \
  --code-ref="libb/config.py:L42@abc123"
```

### 频率限制

每个发送方-接收方对最多 20 条并发待处理消息；每 24 小时总计 50 条。触碰这些限制是重新考虑方式的信号 — 如果你在产生那么多 inbox 消息，你可能在以错误的粒度工作。更宏观的设计问题属于讨论区，不属于单独消息。

## §8. 工作流与知识库 — 记忆不够用的时候

记忆文件是简短的原子断言。不是什么都属于那里。

### 用工作流的时候

存在**带有可执行步骤的可重复过程** — 合并前检查、部署序列、依赖升级流程、事故响应脚本。工作流有一个能跑的 `spine.*`、能验证它的 `fixtures/`、以及跟踪成功率的 `metrics.yaml`。如果你会把一个过程在记忆里写成一列 shell 命令，那它是工作流。

位置：`.memory/workflows/<name>/`

通过 `workflow_ptr` 在记忆中暴露它，让 LLM 能发现它而不必提前加载完整过程。

### 用知识库文章的时候

存在**有人会专门来阅读的大段领域材料** — 架构概述、迁移指南、平台规范参考、安全模型说明。知识库文章是多节文档。LLM 生成的 `_compiled.md` 摘要在完整文章太大无法加载时能高效地进入上下文预算。

位置：`.memory/kb/<topic>/`

### 决策规则

- 写的内容有单一明确断言，可以独立被 supersede → 记忆
- 写的内容有必须运行、有可量化结果的步骤 → 工作流
- 写的内容是参考材料，人们会按章节导航着看 → 知识库
- 写的内容有超过 3 个节，或读起来像教程 → 知识库，不是记忆

写一条 30 行的记忆，一步步描述某个过程，是资产想变成工作流的信号。演化引擎最终会提议升级 — 但你可以从一开始就把它放对地方，省掉这个麻烦。

---

## §9. 什么不该存进记忆

即使用户明确要求存，也要考虑请求是否适合记忆系统。

**不要存：**

1. **代码模式、惯例或文件路径** — 读代码库就能得到。如果某个模式重要到需要记住，它属于知识库文章。
2. **git 历史或近期变更** — `git log` 和 `git blame` 是权威。说"我们在 2026-04-15 合入了 auth 重构"的记忆只是噪声；commit 在那里。
3. **特定 bug 的调试解决方案** — 修复在代码里；上下文在 commit message 里。指向 commit 或 issue 的 `reference` 记忆就足够了。
4. **`CLAUDE.md` / `AGENTS.md` 的内容** — 已经由适配器在会话开始时加载了。把适配器内容复制到记忆里会造成静默覆盖冲突，并膨胀上下文预算。
5. **临时会话状态** — 当前任务、临时决策、你刚修改的文件列表、今天的工作笔记。
6. **近期活动摘要、PR 列表、changelog 条目** — 版本控制系统和 issue 跟踪器是权威。记忆不是 changelog。
7. **SPEC、DESIGN 或其他规范文档中已有的内容** — 改成写一条 `reference` 记忆指向那个文档。

**当用户要求存这类东西时：**

不要拒绝 — 反问："这里面什么是出人意料的或不显然的？" 出人意料的那部分才值得存。其余的已经可以找到了。

示例："存一下我的 PR 列表" — 问："关于这个 PR 列表，我该记住什么？列表本身在 GitHub 里。这些 PR 有什么模式、决策或约束，是以后不会显然可见的？" 他们说出来的那个，就是记忆。

## §10. 随时间演化记忆

### 用 supersede，不要直接改写

当一条记忆是错的或过时了：

**推荐：** 创建一条新记忆，加上 `supersedes: <old-id>`。旧记忆的生命周期转变为 `deprecated`（待退）。历史得以保留；血统可追溯；一致性引擎能跟踪演化。

**避免：** 不创建 supersede 链而直接在原地编辑旧记忆。这会破坏引用，隐藏历史，阻止一致性引擎检测静默覆盖冲突。

**如何 supersede：**

1. 写一条内容已修正的新记忆
2. 在新记忆的 frontmatter 里加上 `supersedes: <old-id>`
3. 运行 `engram memory archive <old-id>`，或让一致性引擎提议转变
4. 更新 `MEMORY.md` 以反映新条目

### 不需要 supersede 的更新

当记忆内容准确但细节过时 — 事实依然成立，但某些细节变了 — 使用 `engram memory update` 刷新内容并更新 `updated` 字段。这不创建 supersede 链，适用于小幅修正、新增限制或更新示例。

### 归档，不要删除

即使一条记忆明显已经过时，也用 `engram memory archive <id>` 而不是直接删除文件。已归档的记忆移到 `~/.engram/archive/`，在物理删除之前保留至少六个月。这不是仪式 — 这是一致性引擎跟踪"我们有意识地停止相信 X"（与"X 从未被记录"不同）的方式。

不经归档直接物理删除记忆文件是协议违规。任何静默这样做的工具不符合规范。

### 当一条记忆有争议时

不要悄悄 supersede 其他资产依赖的记忆。用 `engram consistency scan` 检查入站引用。如果其他记忆或工作流引用了你想改变的资产，先草拟 supersede 记忆，让 `engram review` 在旧的被标为 `deprecated` 之前浮出来供人工确认。

## §11. 记忆与跨 Scope 交互

写之前，思考 scope。正确的问题不是"技术上什么 scope 是对的"，而是"这条规则实际上在什么 scope 成立"。

### 提升 scope

- 适用于你个人参与的每个项目的规则，属于 `user` scope，而不是在每个仓库里重复的 `project` scope 记忆。
- 适用于团队所有项目的规则，属于 `team` scope。一条团队级记忆能省去每个团队成员各自写相同记忆的麻烦。
- 组织已经采纳的规则，属于 `org` scope。

当某个模式在多个项目中反复出现时，这种反复就是考虑提升 scope 的信号。把它提议到相关 scope（添加到 `~/.engram/team/<name>/`，或在团队的记忆 review 中提出来）。

### Pool 订阅

如果某些知识对你的归属层级之外的人也有用 — 其他团队、其他组织或更广泛的社区 — 考虑它是否属于 `pool`。池是按主题共享的知识库，按需订阅。订阅在 `org` 层级的池会把内容传播给组织内的每个项目。订阅在 `user` 层级的池只影响你自己的项目。

### 更高的 scope 要求更高的谨慎

在 `team` 或 `org` scope 写作时，应用更严格的标准：

- `limitations:` 比以往更重要，不是更少——组织级规则影响很多人和很多上下文；意料之外的边界情况更可能出现
- `**Why:**` 必须详细——组织级 scope 的未来读者可能不共享你的上下文
- `enforcement:` 必须慎重设置——组织级的 `mandatory` 是一个强声明

带有 `enforcement: mandatory` 的 `org` scope `feedback` 记忆是 engram 系统中最权威的东西。它优先于每一条 `user`、`team` 和 `project` 记忆。只有当你真的是那个意思时，才那样写。

## §12. 快速审计 Checklist

在完成任何记忆创建或更新之前，核对：

- [ ] 必填 frontmatter 字段存在：`name`、`description`、`type`、`scope`
- [ ] 对于 `feedback`：`enforcement` 已显式设置（非默认）
- [ ] 对于 `project`：所有日期都是绝对 ISO 8601 格式（不是"下周"或"Q2"）
- [ ] Body 包含 `**Why:**` 和 `**How to apply:**` 节（对于 `feedback`、`project`、`agent`）
- [ ] 引用指向真实的路径、URL 或资产 ID（不是过时的或虚构的）
- [ ] 没有重复已有记忆——事先用 `engram memory search "<主题>"` 搜索过
- [ ] Scope 是合理的最紧范围（默认 `project`；只有明显更宽泛时才提升）
- [ ] `description` 不超过 150 字符（它出现在 `MEMORY.md` 中，必须可快速扫读）
- [ ] 没有敏感内容（API 密钥、私密凭据、个人信息），除非 scope 严格本地且访问受控
- [ ] 对于 `agent`：`confidence` 块存在，带有 `source: agent-learned` 或具体的版本引用
- [ ] 对于 supersede 记忆：`supersedes: <old-id>` 已声明，旧记忆已排队归档
- [ ] `MEMORY.md` 将在同一操作中更新（索引与 store 不同步是 bug）

---

记忆纪律是复利的。一条写好的记忆能在一个项目的生命周期内防止数百次重复对话。花那 30 秒选对 scope、写一个真正的 `**Why:**` 子句、慎重地设置 `enforcement:` — 你是在为每一个未来的会话、每一个未来的 LLM、每一个继承这个 store 的未来成员而写。
